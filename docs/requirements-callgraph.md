## 0. スコープ判断の前提（正直な記録）

本要件の「ゴール」について本来ユーザーへ確認したかったが、当セッションでは対話ができ
なかったため、プロジェクトの既存方針から最も整合する解釈を採用した。

- 採用した解釈: **既存アプリ（app.py）に呼び出しグラフ機能を統合し、現状の自前静的解析
  の弱点を実績あるOSSツールで埋める。静的×動的のハイブリッドを踏襲する。**
- 採用の根拠:
  1. ブランチ名が `python-callgraph-tools` であり、調査対象＝コールグラフ生成OSS。
  2. `docs/todo.md`「次にやること」の筆頭が **ファイル横断の呼び出しツリー統合**。
  3. `docs/knowledge.md` に既に「自前 `_ComplexityVisitor` を保守せず radon へ寄せた」
     「静的×動的ハイブリッドで役割分担する」という決定があり、同じ判断を呼び出しグラフ
     にも適用するのが一貫している。
- もし本当のゴールが別（例: OSS選定PoCだけ／独立CLI群の整備）なら、本書の「§7 フェーズ
  計画」以降を差し替えれば足りるよう、選定根拠（§4）とPoC基準（§5）を独立させてある。

## 1. 背景と現状の課題

### 1.1 現状（何が動いているか）

`code-quality-viz` は「テスト×関数 紐付けビューア」であり、`workspace/` 配下の処理側・
テストコードを選ぶと以下を1画面で見せる。

- テスト×関数のマトリクス（紐付け・テスト漏れ・空振りテスト）
- 複雑度（radon）× 実カバレッジ（pytest-cov）の掛け合わせ
- **全関数を呼び出しの親子でぶら下げた呼び出しツリー**（`link_analyzer.py` の
  `_CallCollector` / `_compute_calls` による自前静的解析）

LLM は使わず、静的解析は標準ライブラリ `ast`、動的計測は pytest-cov / coverage、
複雑度は radon という構成（`requirements.txt`）。

### 1.2 課題（今回埋めたい弱点）

自前の呼び出しツリー（`link_analyzer.py:170-216`）には、コード・ドキュメントに明記済み
の原理的限界がある。

| # | 課題 | 根拠（既存記録） |
|---|------|------------------|
| C1 | **ファイル横断のツリー統合ができない**。呼び出しツリーはファイルごとに分割表示。 | `README.md`「できないこと」／`todo.md` 次にやること筆頭 |
| C2 | **動的dispatchを辿れない**。visitorパターンの `self.visit()` 等は静的に解決できず、該当メソッドが親を持たず根として並ぶ。 | `knowledge.md` 2026-07-02 UI刷新 |
| C3 | **同名関数を安全側で保留**。import で由来が確定できない呼び出しは辺を張らない（`len(targets)==1` のみ採用、`_compute_calls`）。 | `link_analyzer.py:213` |
| C4 | 自前解析エンジンが**保守対象**。ダックタイピング・高階関数・別名importの網羅は自前では追い切れず、テスト・保守コストが増える。 | radon 採用と同じ論点（`knowledge.md`） |

### 1.3 調査からの一次結論（要件の土台）

- 静的解析は動的言語では**原理的に不完全**（code2flow READMEの明言）。「完璧なコール
  グラフ」を自前で目指すのは筋が悪い。
- 学術ツール PyCG は高精度（精度~99.2%）だが**アーカイブ済み**で新規採用は不可。
- 実績・活性の高い現行OSS（pyan3・code2flow・pydeps・py-spy・gprof2dot・snakeviz）を
  組み合わせ、**静的で構造を出し、動的で実行経路を裏取りする**のが現実解。
- したがって本要件は「自前解析の拡張」ではなく「**実績OSSへの委譲＋薄い統合層**」を選ぶ
  （radon 導入時と同じ設計判断）。

## 2. 目的とゴール

### 2.1 目的

生成AIが書いた／他人が書いた**長大で見通しの悪いPythonコードを、呼び出し関係で網羅的に
理解する**。とくに現状ツールが取りこぼす **ファイル横断の呼び出し** と **実行経路の裏取り**
を可能にする。

### 2.2 完了条件（ゴール）

1. `workspace/` の複数ファイルを選ぶと、**ファイル横断で統合された関数呼び出しグラフ**が
   見られる（C1 解消）。
2. 静的グラフが取りこぼす経路を、**実行経路（動的）で裏取り**できる（C2 の緩和）。
3. 呼び出しグラフ生成が**実績あるOSSに委譲**され、自前 `_CallCollector` の保守負担が減る
   か、少なくとも自前解析の限界が画面に明示される（C4）。
4. 既存の**マトリクス・複雑度×カバレッジ・気付き**は後方互換で維持される。

## 3. 用語（画面にも凡例として出す）

- **静的コールグラフ**: コードを実行せず、ソースの構文から「誰が誰を呼びうるか」を推定
  した図。全体像に強いが、動的機能で取りこぼす。
- **動的コールグラフ**: 実際にコードを走らせ「今回の実行で実際に何が呼ばれたか」を記録
  した図。通った経路しか出ないが、確実。
- **caller / callee**: 呼ぶ側 / 呼ばれる側。
- **ファイル横断**: あるファイルの関数が別ファイルの関数を呼ぶ関係。現状ツールが未対応。

## 4. 採用ツールの選定（調査からの確定事項）

方針: **1機能=1実績ツール**に委譲。アーカイブ済み・更新停止ツールは新規採用しない。

### 4.1 採用（コア）

| 役割 | 採用ツール | 選定理由（調査） | 出力 |
|------|-----------|------------------|------|
| 関数/メソッド粒度の静的コールグラフ（**ファイル横断**の主役） | **pyan3** | `ast`+`symtable` で defines/uses を解析。**2026年に開発再開**、Python 3.10〜3.14でテスト済み。DOT/SVG/**HTML(pan/zoom)**/**text** を出力し、`--depth`で粒度、`--paths-from/--paths-to`で経路列挙。text出力はAIエージェント供給用と明記。 | DOT/SVG/HTML/text |
| 全体俯瞰の一枚絵・特定関数の周辺切り出し | **code2flow** | ディレクトリ全体を1枚に。`--target-function`＋`--downstream-depth`でスパゲッティを局所展開。★4.6kで実績大。ASTベースで3.12動作見込み。 | DOT→PNG/SVG |
| モジュール依存の俯瞰・循環import検出 | **pydeps** | 最新 v3.0.6(2026-04)で活発。`--cluster`/`--show-cycles`/`--max-bacon`。**関数ではなくモジュール粒度**である点に注意して併用。 | SVG/PNG/DOT |
| 実行経路の裏取り（動的） | **cProfile + gprof2dot** | 標準ライブラリ cProfile の出力を呼び出し回数・時間つきコールグラフへ変換。既存の pytest 実行系（subprocess）と親和。 | DOT→PNG/SVG |
| 実行経路の階層ツリー表示 | **snakeviz** | ブラウザ上の icicle（つらら）＝呼び出しスタックのツリー。ユーザーが求めた「階層ツリー」に最も近い。cProfile出力専用。 | ブラウザ(icicle/sunburst) |

### 4.2 不採用・保留（調査で確認済み）

- **PyCG**: 2023-11-26 アーカイブ。高精度だが新規採用しない。
- **pycallgraph2**: 2019年最終・3.12未検証。動的グラフが必要なら後継 **python-call-graph**
  （3.8〜3.13, 2025-06）を候補にするが、cProfile+gprof2dot を優先。
- **Doxygen**: 多言語・網羅的だがPythonのメソッド追跡に既知課題、生成が重い（neovim Wiki
  で約30分/2.1GB）。本用途にはオーバースペック。
- **tach / pyreverse**: それぞれモジュール境界管理・UMLクラス図で、関数コールグラフとは
  目的が異なる。将来の別要件（クラス設計理解）で検討。
- **Sourcetrail/NumbatUI**: GUI・WIP・要ビルド。WSL2でX11/Wayland前提。本フェーズ対象外。

### 4.3 補助（ツール外）

- **VS Code Call Hierarchy**: 「xxx └yyy」型のドリルダウン。ツールに実装せず、**運用手順
  として README に案内**する（アプリ機能ではなくIDE補完）。

## 5. 機能要件

記法: 【必須】=本フェーズで実装 / 【任意】=実装できれば良い / 【対象外】=明示的に不実装。

### 5.1 静的コールグラフ（ファイル横断）— C1 解消

- FR-1【必須】選択した複数ファイルを対象に、**ファイル横断の静的コールグラフ**を生成・
  表示する。生成は pyan3 を subprocess で呼ぶ（`python -m` 形式でvenv内から確実に起動、
  radon/pytest と同じ流儀 = `knowledge.md` 準拠）。
- FR-2【必須】巨大化対策として、**表示ノードを絞り込める**こと。少なくとも起点関数の指定
  ＋深さ制御（pyan3 `--depth` / `--paths-from` / code2flow `--target-function`
  `--downstream-depth`）のいずれかを提供する。
- FR-3【必須】グラフ画像（SVG/PNG）を画面に表示し、**ファイルへ書き出せる**こと。
- FR-4【任意】pyan3 の **text出力** を「詳細」内に置き、コピーして別のAIエージェントへ
  そのまま渡せるようにする（pyan3の想定用途）。
- FR-5【任意】pydeps による**モジュール依存図**（`--cluster`）と**循環import検出**
  （`--show-cycles`）を別パネルで提供する。

### 5.2 動的コールグラフ（実行経路の裏取り）— C2 緩和

- FR-6【必須】選択したテスト／エントリを **cProfile で実行**し、gprof2dot 経由で
  呼び出し回数つきの実行コールグラフを生成・表示する。既存のフォルダ解析（pytest 実行）
  と同じ subprocess 実行系に載せる。
- FR-7【任意】snakeviz を起動し、**icicle 階層ツリー**をブラウザ表示する導線を置く
  （WSL2ではURL手動オープンになりうる旨を注記）。
- FR-8【必須】静的グラフと動的グラフを**並置**し、「静的にあるが動的で通らなかった経路」
  「動的で通ったが静的に出ていない経路」を人が突き合わせられるようにする（両者は補完
  関係、という調査結論を画面で体現）。

### 5.3 既存機能との統合・後方互換

- FR-9【必須】既存の **マトリクス・複雑度×カバレッジ・気付き** は変更せず維持する。
- FR-10【必須】新機能は既存フォルダ解析フローに**パネル／詳細expanderとして追加**し、
  1ファイル選択時の従来表示を壊さない（`knowledge.md` の後方互換方針）。
- FR-11【必須】自前 `_CallCollector` による現行ツリーの扱いを決める：
  - 方針A（推奨）: 静的コールグラフを **pyan3 に一本化**し、自前ツリーは撤去または
    「簡易版」として詳細へ格納。保守対象を減らす（radon 導入と同じ判断）。
  - 方針B: 自前ツリーを当面残し、pyan3 を「ファイル横断版」として併設。
  - 実装時にどちらを採るか PoC（§6）の結果で確定し、`knowledge.md` に決定を記録する。

### 5.4 明示的な対象外

- FR-X1【対象外】完璧な静的コールグラフ（動的機能の完全追跡）。原理的に不可能。
- FR-X2【対象外】Doxygen 相当の全ドキュメント生成、Sourcetrail 相当のGUI探索。
- FR-X3【対象外】tach/pyreverse によるモジュール境界強制・UMLクラス図（別要件）。
- FR-X4【対象外】任意コードのWeb公開実行（§6.3 セキュリティ）。

## 6. 非機能要件・制約

### 6.1 実行環境（調査の一次確認事項）

- NFR-1 対象: **WSL2 Ubuntu + Python 3.12**（3.9+ で概ね動作）。
- NFR-2 **Graphviz が前提**。導入手順に `sudo apt install graphviz` を明記し、未導入時は
  画面にインストール方法を出す（DOT→画像変換が全静的ツールで必要）。
- NFR-3 **py-spy は本フェーズのコア採用にしない**（実行中プロセスへのアタッチに
  `SYS_PTRACE`／sudo が要る）。将来「動いているプロセスを覗く」要件が出たら別途検討。
- NFR-4 ブラウザ表示系（snakeviz, pydeps, tach --web）は WSL2 でWindowsブラウザが自動
  起動しない場合があるため、**URLを画面に表示**して手動オープンできること。

### 6.2 依存・性能

- NFR-5 追加依存は `requirements.txt` に明記（`pyan3`, `code2flow`, `pydeps`,
  `gprof2dot`, `snakeviz`）。Graphviz は OS 依存のため README で案内。
- NFR-6 大規模コードでグラフが判読不能化したら、§5.1 FR-2 の絞り込みで対処。ノード数が
  一定を超えたら**全体一枚絵の生成を抑止し、起点指定を促す**（調査の判断閾値に対応）。
- NFR-7 生成した一時ファイル（DOT/画像/pstats）は使用後に削除する（既存 coverage.json の
  後始末と同じ流儀）。

### 6.3 セキュリティ（既存方針の継続）

- NFR-8 動的解析＝**選んだテスト/スクリプトを実際に実行する＝任意コード実行**。既存の
  フォルダ解析と同じく **ローカル自用専用**とし、画面に明記。Web公開はしない
  （`knowledge.md` 2026-07-02 の決定を踏襲）。

### 6.4 品質・テスト

- NFR-9 新規統合層は subprocess 呼び出しをラップした薄い関数群とし、**単体テスト**
  （ツール未導入時のフォールバック、DOT解析、絞り込み引数の生成）を追加する。
- NFR-10 `app.py` は **AppTest スモーク**で新パネルが例外なく描画されることを検証
  （既存 AppTest の流儀を踏襲、全テストパス維持）。

## 8. 受け入れ基準（Acceptance Criteria）

- AC-1 `workspace/` の複数ファイルを選び、**別ファイルへの呼び出しが1本の静的グラフ**に
  現れる（C1 が解消していること）。
- AC-2 起点関数と深さを指定して、大きなコードでも**判読可能な部分グラフ**に絞れる。
- AC-3 同一対象で**動的実行グラフ**を生成でき、静的グラフと並べて差分を目視できる。
- AC-4 既存のマトリクス・複雑度×カバレッジ・気付きが**従来どおり**表示される（回帰なし）。
- AC-5 Graphviz 未導入・ツール未導入時に、**落ちずに導入手順を案内**する。
- AC-6 追加した統合層の単体テストと AppTest スモークが**全てパス**する。

## 9. リスクと留意点（調査からの正直な移送）

- R-1 **静的グラフは近似**。ダックタイピング・`getattr`/`eval`・デコレータ・動的import・
  モンキーパッチでエッジの取りこぼし／誤リンクが起きる。画面に限界を明記し、動的で裏取り。
- R-2 **動的グラフは実行した経路のみ**。未実行の分岐は出ない。全体像は静的、確証は動的、
  という補完関係を UI と文言で担保する（FR-8）。
- R-3 **code2flow はリリースが2023年で停止**（★は高い）。コア一本足にせず、ファイル横断
  の主役は開発再開済みの **pyan3** に置く。pyan3 は再開直後ゆえ安定性は PoC で要確認。
- R-4 **pydeps はモジュール粒度**であり関数コールグラフではない。役割を混同しないよう
  パネルを分け、凡例で明示する。
- R-5 星数・リリース日は 2026-07 時点の調査値で変動しうる。採否は PoC の実測で最終確定。
