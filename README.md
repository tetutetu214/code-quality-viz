# テスト×関数 紐付けビューア

`workspace/` に置いた処理側コードとテストコードを選ぶと、**どの処理関数がどのテストクラスに紐づいているか**、**テスト漏れはないか**、**複雑度が高すぎる関数はどれか**を表示するツールです。処理側・テスト側とも複数ファイルを選べます。

LLM は使いません。すべて Python 標準ライブラリの `ast` による静的解析です。

---

## 何ができるのか

- **マトリクス表示**: 行=関数・列=テストクラス・交点に記号（● 直接呼び出し(C) / ○ self 代入経由(B)）。**空の行（薄赤）＝テスト漏れ**、**空の列（薄橙）＝どの関数にも紐づかないテスト**が一目で分かります。複数ファイルで関数名が衝突する場合は `ファイル名.関数名` で区別します。
- サマリに **テスト済み率（カバレッジ%）** を俯瞰表示します。
- 各関数の **複雑さ**（分かれ道の数＝複雑度・行数・引数の数・入れ子の深さ）を一覧で出し、**数値が大きいと何が悪いのか**を専門用語なしで説明します。注意ラインを超えた関数は「評価 ⚠ 要注意」で示します。
- テスト漏れ・空振りテスト・複雑すぎ・テスト不足などを **気付き** として平易な文章で列挙します。
- **ファイル横断コールグラフ**: 選んだ全ファイルを横断して「どの関数がどの関数を呼ぶか」を 1 枚のツリー＋図にまとめます（別ファイルへの呼び出しも 1 本につながります）。実績ある OSS の [pyan3](https://github.com/Technologicat/pyan) に委譲し、同名関数もスコープ解決で区別します。起点の関数とたどる深さで絞り込めます。

### なぜマトリクスなのか（設計の根拠）

多対多で密な紐付けをノードリンク図（線でつなぐ二部グラフ）で描くと線が交差して読めなくなります（hairball 問題）。隣接マトリクスは各紐付きに固有のセルがあるため交差が原理的に起きず、特定ペアの有無を交点で即確認できます。詳しい根拠は `SOW.md` を参照してください。

### 紐付けの根拠（2種類）

- **(C) 直接呼び出し**: テスト本体で処理関数を素の名前で直接呼んでいる（例: `update_zip_list_txt('bucket', 'folder/A')`）。
- **(B) self 代入経由**: `setUp` などで `self.target_func = check_gcs_zip_http` と代入し、`self.target_func(...)` で呼んでいる。`from main import main as main_func` のような import の別名も元の関数名へ読み替えて解決します。

### できないこと（仕様）

- 上部の「全関数ツリー」（自前の静的解析）はファイルごとに表示します。**別ファイルへの呼び出しまで 1 枚に統合したツリー／図は、その下の「ファイル横断コールグラフ」（pyan3）**で見られます。
- 静的解析である以上、`getattr` / `eval` / デコレータ / 動的 import / visitor の `self.visit()` のような**動的な呼び出しは辿れません**（該当関数は親を持たず入口として並びます）。実際に通ったかは上のカバレッジ（動的計測）で裏取りしてください。
- `self.x = 名前` の代入は **一段だけ** 解決します。多段の変数受け渡しは追いません。
- 処理側に同名の関数が複数ある場合は、import 文でファイル由来を解決できる場合だけ紐付けます。判別できない場合は紐付けを保留し、画面下に「保留」として表示します。
- モックの差し替えや、文字列から関数名を組み立てる呼び出しは追えません。
- 複雑度のしきい値（複雑度10 / 行数50 / 引数5 / ネスト4）はあくまで目安です。

## 必要なもの

- Python 3.12（3.9 以上でおそらく動作します）
- Python パッケージ（`requirements.txt`）: Streamlit / pytest / pytest-cov / coverage / radon / pyan3。
  紐付け解析そのものは標準ライブラリ `ast` のみ。カバレッジ計測は pytest 系、複雑度は radon、
  ファイル横断コールグラフは pyan3 を、いずれも subprocess で呼びます。
- **Graphviz（OS 側）**: ファイル横断コールグラフの**図を SVG で保存**するのに必要です。
  `sudo apt install graphviz`（Debian/Ubuntu）/ `brew install graphviz`（macOS）。
  ブラウザ上での図表示だけなら Graphviz の導入は不要です（未導入時は画面に導入手順を出します）。

## ファイル構成

```
code-quality-viz/
├── app.py                   ← 画面（これを実行する）
├── link_analyzer.py         ← 静的解析エンジン（紐付け・複雑度・ファイル単位の呼び出しツリー）
├── cross_check.py           ← 動的解析層（pytest 実行→カバレッジ×複雑度の突き合わせ）
├── callgraph.py             ← ファイル横断コールグラフ層（pyan3 委譲）
├── test_link_analyzer.py    ← 静的解析エンジンの単体テスト
├── test_cross_check.py      ← 動的解析層の単体テスト
├── test_callgraph.py        ← コールグラフ層の単体テスト
├── test_app.py              ← 画面（AppTest スモーク）
├── requirements.txt         ← 依存（streamlit / pytest 系 / radon / pyan3）
├── README.md                ← この説明
├── SOW.md                   ← 可視化形式の選定根拠（情報可視化研究の調査）
└── docs/                    ← 設計判断・知見の記録
```

`app.py` は `link_analyzer.py` / `cross_check.py` / `callgraph.py` が同じフォルダにあることを前提にします。

> **補足（IDE 併用）**: 呼び出し元→呼び出し先を対話的にたどりたいときは、VS Code の
> **Call Hierarchy**（関数を右クリック →「Show Call Hierarchy」/ `Shift+Alt+H`）が手軽です。
> 本ツールは全体像の俯瞰、IDE は 1 箇所の深掘り、と使い分けると効率的です。

## インストールと実行

```bash
cd /path/to/code-quality-viz
pip install -r requirements.txt   # 実体は streamlit のみ
streamlit run app.py
```

Cloud Shell など、起動時に `Rejecting WebSocket connection from disallowed origin` が出る環境では、次のオプションを付けてください（自分専用の一時確認時のみ）。

```bash
streamlit run app.py --server.enableCORS false --server.enableXsrfProtection false
```

## 使い方

1. `workspace/` に処理側コードとテストコードを置き、処理側ファイル・テストファイルを選びます。
2. 「解析する」を押します。
3. **マトリクス**で紐付きを確認します。空の行（薄赤）＝テスト漏れ、空の列（薄橙）＝空振りテストが一目で分かります。行の並べ替えも選べます。
4. 「関数の複雑さチェック」で、複雑すぎる関数を確認します（上ほど複雑）。表の上にある「この表の見方」を開くと、各項目の意味と何が悪いのかが読めます。
5. 「気付き」で、テスト漏れ・空振りテスト・複雑すぎ・テスト不足の指摘を確認します。
6. サイドバーの「書き出し」から、両方のコードをまとめて保存できます。

## 解析エンジンのテスト

```bash
python3 -m unittest test_link_analyzer -v
```

証拠B / 証拠C の紐付け、import 別名の解決、複雑度計測、テスト漏れ検出を検証しています。
