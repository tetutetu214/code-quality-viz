# テスト×関数 紐付けビューア

処理側コード（`main.py` 相当の1ファイル）と、それに対する unittest 形式のテストコードを貼ると、**どの処理関数がどのテストクラスに紐づいているか**、**テスト漏れはないか**、**複雑度が高すぎる関数はどれか**を静的に表示するツールです。

LLM は使いません。すべて Python 標準ライブラリの `ast` による静的解析です。

---

## 何ができるのか

- **マトリクス表示**: 行=関数・列=テストクラス・交点に記号（● 直接呼び出し(C) / ○ self 代入経由(B)）。**空の行（薄赤）＝テスト漏れ**、**空の列（薄橙）＝どの関数にも紐づかないテスト**が一目で分かります。セルにマウスを載せると行と列が強調されます（クロスヘア）。行の並べ替え（テスト数が少ない順／グループ順／名前順）も選べます。
- サマリに **テスト済み率（カバレッジ%）** を俯瞰表示します。
- 各関数の **複雑さ**（分かれ道の数＝複雑度・行数・引数の数・入れ子の深さ）を一覧で出し、**数値が大きいと何が悪いのか**を専門用語なしで説明します。注意ラインを超えた関数は「評価 ⚠ 要注意」で示します。
- テスト漏れ・空振りテスト・複雑すぎ・テスト不足などを **気付き** として平易な文章で列挙します。
- 処理側コードとテストコードをまとめて1ファイルに保存・読込できます。

### なぜマトリクスなのか（設計の根拠）

多対多で密な紐付けをノードリンク図（線でつなぐ二部グラフ）で描くと線が交差して読めなくなります（hairball 問題）。隣接マトリクスは各紐付きに固有のセルがあるため交差が原理的に起きず、特定ペアの有無を交点で即確認できます。詳しい根拠は `SOW.md` を参照してください。

### 紐付けの根拠（2種類）

- **(C) 直接呼び出し**: テスト本体で処理関数を素の名前で直接呼んでいる（例: `update_zip_list_txt('bucket', 'folder/A')`）。
- **(B) self 代入経由**: `setUp` などで `self.target_func = check_gcs_zip_http` と代入し、`self.target_func(...)` で呼んでいる。`from main import main as main_func` のような import の別名も元の関数名へ読み替えて解決します。

### できないこと（仕様）

- 別ファイルの処理は追えません（処理側は1ファイル前提）。
- `self.x = 名前` の代入は **一段だけ** 解決します。多段の変数受け渡しは追いません。
- 処理側に同名の関数が複数ある場合（別クラスの同名メソッドなど）は、どちらか判別できないため紐付けを保留し、画面下に「保留」として表示します。
- モックの差し替えや、文字列から関数名を組み立てる呼び出しは追えません。
- 複雑度のしきい値（複雑度10 / 行数50 / 引数5 / ネスト4）はあくまで目安です。

## 必要なもの

- Python 3.12（3.9 以上でおそらく動作します）
- Streamlit（このツールが使う唯一の外部ライブラリ。解析自体は標準ライブラリのみ）

---

## ファイル構成

```
code-quality-viz/
├── app.py                   ← 画面（これを実行する）
├── link_analyzer.py         ← 解析エンジン（app.py が内部で使う）
├── test_link_analyzer.py    ← 解析エンジンの単体テスト
├── requirements.txt         ← 依存（streamlit）
├── README.md                ← この説明
├── SOW.md                   ← 可視化形式の選定根拠（情報可視化研究の調査）
└── docs/                    ← 設計判断・知見の記録
```

`app.py` だけでは動きません。`link_analyzer.py` が同じフォルダに必要です。

---

## インストールと実行

```bash
cd /path/to/code-quality-viz
pip install -r requirements.txt   # 実体は streamlit のみ
streamlit run app.py
```

Cloud Shell など、起動時に `Rejecting WebSocket connection from disallowed origin` が出る環境では、次のオプションを付けてください（自分専用の一時確認時のみ）。

```bash
streamlit run app.py --server.enableCORS false --server.enableXsrfProtection false
```

---

## 使い方

1. 左の欄に処理側コード、右の欄にテストコードを貼ります。
2. 「解析する」を押します。
3. **マトリクス**で紐付きを確認します。空の行（薄赤）＝テスト漏れ、空の列（薄橙）＝空振りテストが一目で分かります。行の並べ替えも選べます。
4. 「関数の複雑さチェック」で、複雑すぎる関数を確認します（上ほど複雑）。表の上にある「この表の見方」を開くと、各項目の意味と何が悪いのかが読めます。
5. 「気付き」で、テスト漏れ・空振りテスト・複雑すぎ・テスト不足の指摘を確認します。
6. サイドバーの「書き出し」から、両方のコードをまとめて保存できます。

---

## 解析エンジンのテスト

```bash
python3 -m unittest test_link_analyzer -v
```

証拠B / 証拠C の紐付け、import 別名の解決、複雑度計測、テスト漏れ検出を検証しています。
